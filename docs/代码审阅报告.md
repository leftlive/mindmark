## 主要问题

### P1 切换到思维导图的命令调用了错误的方法，导致从 Markdown 无法正确切到 MindMap

- 位置：
  - `src/main.ts:61-63`
  - `src/main.ts:76-77`
  - `src/main.ts:1246-1267`
- 说明：
  - `Toggle to markdown or mindmap` 在当前是 Markdown 视图时，调用的是 `setMarkdownView()`，不是 `setMindMapView()`。
  - `Set to mindmap view` 也同样错误地调用了 `setMarkdownView()`。
  - `setMarkdownView()` 会显式把叶子切回 `"markdown"`，因此这两个命令在 Markdown 场景下不会进入思维导图视图。
- 影响：
  - 核心切换命令行为与名称不一致。
  - 用户会遇到“命令可执行但视图不切换”的问题。

### P1 事件解绑拼写错误，双击监听会残留，重复打开/刷新后可能触发多次编辑

- 位置：
  - `src/mindmap/mindmap.ts:385`
  - `src/mindmap/mindmap.ts:418`
- 说明：
  - 注册时使用的是 `dblclick`。
  - 解绑时写成了 `dblClick`。
  - 浏览器事件名大小写敏感，导致双击监听不会被正确移除。
- 影响：
  - 视图重建后可能累积旧监听。
  - 双击节点可能出现重复进入编辑、重复逻辑执行和潜在内存泄漏。

### P1 文件联想弹层挂到 `document.body`，但没有销毁路径，会产生长期泄漏

- 位置：
  - `src/mindmap/FileSuggest.ts:13-25`
  - `src/mindmap/mindmap.ts:95-97`
  - `src/MindMapView.ts:361-368`
- 说明：
  - `FileSuggest` 构造时直接把 DOM 节点 append 到 `document.body`。
  - `MindMap`/`MindMapView` 关闭时只清理主视图 DOM 和事件，没有调用任何 `destroy()` 或 `remove()`。
- 影响：
  - 每次创建新 MindMap 实例都会遗留一个建议框 DOM。
  - 长时间使用、多次切换文件后会累积无主节点和状态。

### P2 “导出 HTML” 实际写出的是 PNG 的 `<img>` 标签，不是可独立使用的 HTML 文档

- 位置：
  - `src/MindMapView.ts:67-126`
- 说明：
  - `exportToSvg()` 名称与行为不一致：
    - 先 `toPng()`
    - 再把生成的 data URL 塞进 `img.outerHTML`
    - 最终写入 `.html`
  - 结果文件只是一个单独的 `<img>` 标签片段，不是完整 HTML，也不是 SVG。
- 影响：
  - 命令名、方法名、输出格式三者不一致。
  - 调用方和用户都会误判导出内容。

### P2 文件联想只写入 `[[basename]]`，重名文件场景会生成歧义链接

- 位置：
  - `src/mindmap/FileSuggest.ts:52-58`
  - `src/mindmap/FileSuggest.ts:71`
  - `src/mindmap/FileSuggest.ts:134`
- 说明：
  - 搜索按 `file.basename` 匹配。
  - 插入时也只写 `[[${file.basename}]]`。
  - 在 Obsidian 中同名文件很常见，这会让链接解析结果依赖当前上下文，甚至指向错误文件。
- 影响：
  - 双链自动补全在多目录/同名笔记环境下不可靠。

### P2 工程元数据仍是 sample plugin，占位信息未清理

- 位置：
  - `package.json:2-12`
  - `manifest.json:2-9`
- 说明：
  - `manifest.json` 已经是实际插件信息。
  - 但 `package.json` 仍保留 `obsidian-sample-plugin`、sample description、空 author。
- 影响：
  - 容易造成发布、自动化脚本、仓库识别和依赖审计时的信息混乱。
